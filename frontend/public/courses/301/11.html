UNIFIED MODELLING LANGUAGE

It is a de-facto s/w industry standard modeling language for visualizing, specifying, constructing and documenting system elements in general and s/w systems in particular.

UML is a language for documenting models. It has its syntax and semantics. It provides a set of basic graphical notations that can be combined in certain ways to document the design and analysis results.

A model is an abstraction of a real problem or situation and is constructed by leaving out unnecessary details. This reduces the problem complexity and makes it easy to understand the problem.

It has a well defined syntax and semantics, provides a rich set of graphical artifacts which help in the top-down refinement of object-oriented systems from the phases of requirements capture to the deployment of s/w components. Typically used in an Object-Oriented SDLC model including object-oriented analysis and design phases.

In UML, systems can be modeled by considering 2 aspects :

  1. Behavioral
  1. Structural
  1. Architectural

Each aspect is concerned with both static and dynamic views of the system. The static view represents a projection on the static structures of the complete system description. The dynamic view represents a projection on the dynamic behavior of the system. Views are communicated using a number of diagrams that contain information which emphasize a particular aspect of the system. The RUP model for s/w development uses UML as a tool for the visual modeling of s/w artifacts.

Aspects

While modeling s/w systems, 3 aspects of such systems are considered :

  1. Behavioral Aspects : involve the functionalities provided by the system (static behavioral aspects) and their associated interactions, timings and orderings (dynamic behavioral aspects).
  1. Structural Aspects : involve the class structures of the Object-oriented s/w and the inter-relationships among the classes ( static structural aspects ) and the algorithmic details of the object life cycles and their interactions (dynamic structural aspects).
  1. Architectural Aspects : related to the deployment and distribution of s/w and h/w across the various distributed system components.

Views

Aspects of the system are covered by various static and dynamic views :

  1. Use Case View : focuses on the behavioral aspects of the system and reflects the user’s concerns and requirements. Considers the system as a black box (where the internal working of the system is not known/visible to the user). Users, testers and requirement analysts are interested in this view.
  1. Design and Process View : are the concern of the development team including the analysts, designers and implementers. Design view is concerned with objects, classes and collaborations. The process view is concerned with s/w architecture issues such as concurrency, multithreading, synchronization, performance and scalability issues.
  1. Implementation View : describes the files and components needed to assemble and release the system.
  1. Deployment View : describes the ways to distribute, deliver and install system files and components across the distributed nodes of the system.

Diagrams

Contain model elements such as classes, objects, nodes, components and various types of relationships among them described by generally standardized graphical symbols. A diagram can be used to describe certain system aspects at the different levels of abstraction. Ex. state diagrams can describe system level i/o interactions and can also be used to describe/show state changes of a particular system object across multiple use cases.

UML diagrams are clustered under

  1. Behavioral modeling diagrams
  • Use Case Diagram
  • > Describes the functions of the system and its external users and defines their interfaces.
  • > Shows a number of actors, external to the system and their connection to the use cases representing the services provided by the system.
  • > Use cases can be inherited by other use cases allowing for the reusability of use cases. Can also be associated using the include relationship. This allows for the modularization of use cases. They can similarly be associated with the extend relationship to deal with certain exceptions or abnormal conditions that disrupt the normal execution of the use case.
  • > Is used to model the static behavioral aspects of the use case view of the system to model.
  • > Mainly used to capture and document functional s/w requirements.
  • Statechart Diagram
  • > Represents the possible behavior of an object using state changes.
  • > Can also be used as a system level diagram which shows the system state changes during the operation of the system.
  • > Can be used to model both the dynamic behavioral aspects of the use case view and the dynamic structural aspects of the design and process view of the system to model.
  • Sequence Diagram
  • > Show the interaction between a number of objects and their time ordering.
  • > Can be used to model the dynamic behavioral aspects of the use case view if the interacting objects are interface objects or actors.
  • > In case the interacting objects are internal ones, it can be used to model the dynamic structural aspects of the design and process views of the system to model.
  • Activity Diagram
  • > Shows a sequential or concurrent flow from one activity to another.
  • > Consists of action states, activity states and transitions b/w them.
  • > Usually spans more than one use case or functionality.
  • > Can be used to model both the dynamic behavioral aspect of the use case view and the dynamic structural aspect of the design view of the process to model.
  • Communication Diagram
  • > Shows the objects and the numbered messages they exchange. These messages are numbered to provide a time ordering of the messages.
  • > Can be used to model the dynamic behavioral aspects of the use case view in addition to the dynamic structural aspect of the process view of the system to model.
  • > Communication + Sequence Diagrams = Interaction Diagram, can be easily produced from each other.
  • Timing Diagram
  • > Merges a sequence and state diagram to provide a timeline showing the changes of state of an object over time.
  • > Also shows the messages or interactions exchanged as well as the time and duration constraints attached to the interactions.
  • Interaction Overview Diagram
  • > Is a variation of activity diagram in which the activities represent an activity diagram.

  1. Structural modeling diagrams
  • Class Diagrams
  • > Used to model the static structural aspects of the design and process views of the system to model.
  • > Show the static structure of classes and their possible relationships in the system.
  • Object Diagram
  • > Variant of class diagrams, they use an almost identical notation.
  • > Can be an example of a class diagram that shows possible instantiations of classes during system execution.
  • > Used to model the static structural aspects of the design and process views of the system to model.
  • Composite Diagrams
  • > Shows the internal composition of a class.
  • > Also includes the interaction points (also called ports) through which a component of a class interacts with the other parts of the system.
  • Component Diagram
  • > Shows the physical structure of the code in terms of code components. Components can be composed of source code, a binary or an executable component.
  • > A component contains information about the logical class or the class it implements. This creates a mapping from the design view to the component view.
  • > It is used to model the static architectural aspects of the implementation view of the system to model.

  1. Architectural modeling diagrams
  • Package Diagrams
  • > Shows the existing packages in s/w and the relationship b/w them.
  • > A package can contain components, class diagrams and use case diagrams among other UML diagrams.
  • > Packages can be composed of other packages and 2 packages can be merged.
  • Deployment Diagram
  • > Shows the physical architecture and distribution of the s/w and h/w components in the system.
  • > Used to model the static architectural aspects of the deployment view of the system.

In almost every design methodology, the models of the problem are first constructed. These are called the analysis models. Once the required models of a system have been constructed, these can be used for a variety of purposes during s/w development, including :

  1. Analysis
  1. Specification
  1. Design
  1. Coding
  1. Visualization and understanding of the implementation
  1. Testing

The different UML diagrams provide different perspectives of a s/w system to be developed which facilitates a comprehensive understanding of the system. UML diagrams help to capture the 5 views/models of a system :

  1. User’s view

Captures the functionalities offered by the system to its users. It is a black-box view of the system where the internal structure, the dynamic behaviour of different system components, the implementation etc. are ignored. It is a functional model. Is considered as the central view.

  1. Structural view

Defines the structure of the problem in terms of the kinds of objects important to the understanding of the working of a system and to its implementation. Also captures the relationships among the classes.

  1. Behavioural view

Captures how objects interact with each other in time to realise the system behaviour. The system behavior captures the time-dependent behaviour of the system.

  1. Implementation view

Captures the important components of the system and their interdependencies.

  1. Environmental view

Models how the different components are implemented on different pieces of h/w.
